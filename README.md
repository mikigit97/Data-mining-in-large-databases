## Task 1 — Baby Names Explorer

**Dataset:** US Social Security Administration baby names, 1880–2014 (~1.8 M rows)  
**Backend:** SQLite via the Python `sqlite3` package

The task covers loading a large CSV dataset into a relational database, designing an
efficient schema with indexes, building an interactive Streamlit app on top of it, and
discovering non-obvious patterns from the data using SQL.

---

### Task 1.1 — Data Loading & Schema Design

**Deliverable:** [task1/task1_baby_names.ipynb](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task1/task1_baby_names.ipynb)  
**Supporting file:** `task1/baby_names.db` (generated by running the notebook)  
**README:** [task1/docs/readme_task1_1.md](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task1/docs/readme_task1_1.md)

The notebook builds a reproducible data pipeline:
1. Reads `NationalNames.csv` (~1.8 M rows) in 50 000-row chunks using `pandas read_csv(chunksize=...)` to keep memory usage flat.
2. Creates `baby_names.db` from scratch (deleted and recreated on each run for reproducibility).
3. Loads all rows into the `national_names` table with `PRAGMA synchronous = OFF` + `journal_mode = MEMORY` for a ~10× bulk-insert speedup.
4. Creates three indexes and measures their effectiveness with `EXPLAIN QUERY PLAN` and a timed benchmark.

**Schema:**
```sql
CREATE TABLE national_names (
    Id      INTEGER PRIMARY KEY,
    Name    TEXT    NOT NULL,
    Year    INTEGER NOT NULL,
    Gender  TEXT    NOT NULL,
    Count   INTEGER NOT NULL
);
-- idx_nat_name (Name)
-- idx_nat_year (Year)
-- idx_nat_name_year (Name, Year)  ← covering index
```

The composite `(Name, Year)` index is a covering index for the relative-popularity query —
SQLite's `EXPLAIN QUERY PLAN` confirms it never touches the main table. The benchmark measures
a **50–80× speedup** (150–200 ms → 2–5 ms) for `WHERE Name = ?` queries on 1.8 M rows.

### Task 1.2 — Interactive Name Explorer App

**Deliverable:** [task1/app.py](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task1/app.py)  
**[Live App →](https://data-mining-in-large-databases-kuabme5kqccwyexedxh3mk.streamlit.app/)**  
**README:** [task1/docs/readme_task1_2.md](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task1/docs/readme_task1_2.md)

A three-tab Streamlit web app for exploring 134 years of US baby-name data.

**Tab A — Name Popularity Over Time**  
Enter one or more names (comma-separated) and see an interactive line chart across years.
Toggle between raw birth count and percentage of all births that year — the latter corrects
for US population growth and reveals genuine popularity shifts.

**Tab B — Custom SQL Query Panel**  
A free-text SQL editor with five pre-built example queries:
- Top 10 names in 2010
- Gender-neutral names (balanced M & F usage)
- Names that disappeared after 1950
- Top name per decade (using `ROW_NUMBER() OVER PARTITION BY`)
- Name diversity per year

All queries are safety-checked (SELECT-only, leading comment lines skipped) before execution.
Results are rendered as a sortable table; if the result contains a numeric and a categorical
column, an automatic chart section appears with axis selectors and a Bar / Line toggle.

**Tab C — Name Insights**
- *Name Diversity Over Time* — unique names per year by gender, cached with `@st.cache_data`.
- *Peak Decade Finder* — enter any name to see its decade-by-decade popularity breakdown.

## Task 2 — Oscar Actor Explorer

**Dataset:** The Oscar Award — 11,110 nominations, 97 ceremonies, 1927–2024  
**Backend:** PonyORM 0.7.19 on Python 3.11

The task covers designing a normalised multi-entity ORM schema, loading a flat CSV into it,
building an actor profile app that combines database queries with a live Wikipedia API, and
deriving three interesting findings from the data.

---

### Task 2.1 — ORM Data Modeling

**Deliverable:** [task2/task2_oscar.ipynb](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task2/task2_oscar.ipynb) *(kernel: `oscar-py311`)*  
**Supporting file:** `task2/oscar.db` (generated by running the notebook)  
**README:** [task2/readme_task2_1.md](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task2/readme_task2_1.md)

The notebook defines five normalised PonyORM entities and loads 11,110 nomination rows from
the flat CSV into a relational schema:

| Entity | PK | Notes |
|---|---|---|
| `Ceremony` | `number` | 97 ceremonies, natural PK |
| `Category` | auto | 118 raw category names → 58 canonical |
| `Person` | auto | 6,893 distinct nominees |
| `Film` | auto | composite unique `(title, year)` |
| `Nomination` | auto | FK to all four above; `person` and `film` are `Optional` |

All DB queries use PonyORM generator expressions (`select(n for n in Nomination if ...)`).
No raw SQL is used anywhere in Tasks 2 or 3.

**Note on Python version:** PonyORM 0.7.19 is incompatible with Python 3.14 (the default on Streamlit Cloud). A dedicated Python 3.11 virtual environment (`.venv311/`) is required. The notebook must be run with the `oscar-py311` kernel.

**Deliverable:** [task2/app2.py](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task2/app2.py)

Search any Academy Award nominee by name and see a rich profile card combining Oscar

**Profile card sections:**

## Task 3 — Pokémon Battle Arena

**Dataset:** 800 Pokémon (Gen 1–6) + full 18×18 type-effectiveness matrix  
**Backend:** PonyORM 0.7.19 on Python 3.11 (same `.venv311` as Task 2)

The task covers schema design for a game database, building a playable turn-based battle app,
implementing cheat codes as real database write operations, and analysing the dataset for
statistical insights.

---

### Task 3.1 — Data Loading & Schema Design

**Deliverable:** [task3/task3_pokemon.ipynb](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task3/task3_pokemon.ipynb) *(kernel: `oscar-py311`)*  
**Supporting file:** `task3/pokemon.db` (generated by running the notebook)  
**README:** [task3/readme_task3_1.md](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task3/readme_task3_1.md)

The notebook defines three PonyORM entities, loads 800 Pokémon from `Pokemon.csv`, seeds the
full 18×18 type-effectiveness matrix (324 ordered pairs including all neutral matchups), and
writes a `BattleLog` table for recording game results.

Key schema decisions: `Name` is used as the Pokémon primary key (Pokédex `#` is non-unique
due to Mega forms); `type2` uses `Optional(str, nullable=True)` to store SQL `NULL` for
single-type Pokémon; all 324 type pairs are stored explicitly so the battle app can do a
single `dict.get()` lookup with no Python-side fallback.

---

### Task 3.2 — Battle Game

**Deliverable:** [task3/app3.py](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task3/app3.py)  
**[Live App →](https://data-mining-in-large-databases-vaewz6c7ktlvxwhjdtezxz.streamlit.app/)**  
**README:** [task3/readme_task3_2.md](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task3/readme_task3_2.md)

A fully playable turn-based battle game where all stats and type matchups come from the database.

**Three phases:**

1. **Team Setup** — Searchable multiselect to pick 1–3 Pokémon from all 800. Live preview shows HP, ATK, DEF, Speed, Total, and legendary flag. AI drafts its team randomly from the remaining pool.

2. **Battle** — Side-by-side layout with HP progress bars. Each **Attack!** click resolves one round using:
   ```
   damage = max(1, floor(attacker.attack / defender.defense × 40 × type_mult))
   ```
   Turn order is determined by Speed. Dual-type defenders chain both type multipliers.

3. **Results** — Win/loss banner, turn-by-turn log, and battle history table. One `BattleLog` row is written to the DB at the end of each battle.

### Task 3.3 — Cheat Codes

**Deliverable:** [task3/app3.py](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task3/app3.py) (cheat code section within the same battle app)  
**README:** [task3/readme_task3_3.md](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task3/readme_task3_3.md)

Five cheat codes entered via an expander during battle — each one performs a **real PonyORM
write operation** on `pokemon.db` and immediately updates the live session state:

| Code | DB Operation | Effect |
|---|---|---|
| `UPUPDOWNDOWN` | UPDATE | Doubles HP for each player Pokémon |
| `GODMODE` | UPDATE | Sets active Pokémon's DEF and SP.DEF to 999 |
| `STEAL` | INSERT | Copies the AI's highest-Total Pokémon to the player's team |
| `NERF` | UPDATE | Halves all six stats for every AI Pokémon |
| `LEGENDARY` | INSERT | Adds custom "UltraMewtwo" (780 Total, Psychic/Fighting) to the player's team |

A **Cheat Audit** section on the results screen queries the DB for anomalous values
(DEF ≥ 999, HP > 255, `"Stolen %"` names, `"UltraMewtwo"`) to detect which cheats
were used across all past sessions.

### Task 3.4 — Pokémon Analysis

**Deliverable:** [task3/task3_pokemon.ipynb](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task3/task3_pokemon.ipynb) *(analysis cells at the end of the same notebook)*  
**README:** [task3/readme_task3_4.md](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task3/readme_task3_4.md)

Two insights derived from PonyORM `avg()` and `count()` aggregation queries:

- **Power creep is real:** non-legendary Pokémon average ~+7–10 Total per generation (Gen 1→6), confirmed by `numpy.polyfit` linear regression. A Gen 6 starter has roughly the same base Total as a twice-evolved Gen 1 Pokémon.
- **Dragon-types dominate by design:** Dragon's sky-high average Total is explained by Game Freak placing pseudo-legendaries and box legendaries in the Dragon category by lore convention — not by type mechanics. Removing legendaries brings Dragon's average close to Water and Psychic.

## Task 4 — SQL Detective Agency

**Purpose:** teach SQL to beginners through a story-driven detective game  
**Backend:** SQLite via the Python `sqlite3` package

The task covers designing a pedagogically ordered learning game, persisting progress to a
database, and implementing a suite of engagement features to motivate the learner.

---

### Task 4.1 — Core Platform

**Deliverable:** [task4/app4.py](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task4/app4.py)  
**[Live App →](https://data-mining-in-large-databases-3acrtu4hpzvegfbgfxpddg.streamlit.app/)**  
**Supporting modules:** [db_setup.py](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task4/db_setup.py), [levels.py](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task4/levels.py), [progress.py](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task4/progress.py), [ui_components.py](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task4/ui_components.py), [styles.py](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task4/styles.py)  
**README:** [task4/readme_task4_1.md](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task4/readme_task4_1.md)

The learner plays a detective solving cases by writing real SQL queries against a crime
database. Each case teaches exactly one SQL concept in pedagogical order:

| Case | Concept | Task |
|---|---|---|
| 1 — Open the Case Files | `SELECT *` | Retrieve all 25 suspects |
| 2 — Narrow the Suspects | `WHERE` | Filter to the Docklands district |
| 3 — Top Suspects | `ORDER BY / LIMIT` | Rank suspects by motive score |
| 4 — Crime by District | `GROUP BY / COUNT` | Count crimes per district |
| 5 — Connect the Evidence | `JOIN` | Link suspects to physical evidence |
| Bonus — The Mastermind | `HAVING` | Find suspects with multiple evidence records |

Levels unlock sequentially. Progress is persisted to a `detective_log` SQLite table (UUID
session ID) so refreshing the page restores completed levels, points, and badges.

**Answer checking:** the learner's query is executed against the real DB and the resulting
DataFrame is compared by row count, column set, and key cell values — never by SQL string
matching. Every level has a `diagnose_fn` that returns a specific message explaining *what*
is wrong (e.g. "25 rows returned — you need a WHERE clause"), not just "incorrect".

### Task 4.2 — Engagement & Creativity

**Deliverable:** [task4/app4.py](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task4/app4.py) (engagement features layered on top of the Task 4.1 core)  
**README:** [task4/readme_task4_2.md](https://github.com/mikigit97/Data-mining-in-large-databases/blob/main/task4/readme_task4_2.md)

Five engagement systems woven into the same app:

**1. Detective Rank System**  
Points awarded per solve (15 / 10 / 5 based on number of attempts) unlock five ranks
from *Rookie Constable* to *Master Detective*. Rank-ups fire `st.balloons()`.

**2. Badge System**  
Six badges earned for specific achievements: first case closed, any level with zero hints,
first-try solve, persistence after using both hints, all five main cases complete, bonus level complete. Badges fire `st.toast()` on earn and are displayed as a shelf in the sidebar.

**3. Hot Streak Tracker**  
Consecutive first-try no-hint solves increment a streak counter. At 3+, an amber
"🔥 You're on fire!" banner appears in the sidebar.

**4. Visual SQL Teaching Aids**
- Level 3: before/after sort comparison — two side-by-side tables showing unsorted vs `ORDER BY motive_score DESC`, making the impact of `ORDER BY` immediately visible.
- Level 5: side-by-side `suspects ↔ evidence` JOIN diagram annotated with the join key `suspect_id`.
- Level 2: matching WHERE rows highlighted yellow using `pandas Styler`.

**5. Floating Detective Avatar**  
A fixed-position detective emoji (🕵️) with a speech bubble reacts to game events in real time: greets the player by name, congratulates level completions, announces badge unlocks, and explains SQL errors and wrong answers in detective-voice templates. Implemented via `streamlit.components.v1.html` (parent-document JavaScript injection) to achieve true viewport-fixed positioning — bypassing Streamlit's scrollable container limitation.

## Running Locally

All four apps require Python 3.11 (PonyORM is incompatible with Python 3.14).
A shared virtual environment is used across Tasks 2–4.

```bash
# Clone the repository
git clone https://github.com/mikigit97/Data-mining-in-large-databases.git
cd "Big Data scraping"

# Create the Python 3.11 venv (once)
python3.11 -m venv task2/.venv311
task2/.venv311/Scripts/pip install streamlit pandas plotly ponyorm requests

# Task 1
streamlit run task1/app.py

# Task 2 — run task2/task2_oscar.ipynb first to generate oscar.db
task2/.venv311/Scripts/streamlit run task2/app2.py

# Task 3 — run task3/task3_pokemon.ipynb first to generate pokemon.db
task2/.venv311/Scripts/streamlit run task3/app3.py

# Task 4
task2/.venv311/Scripts/streamlit run task4/app4.py
```
